# 产品体验差异复盘

## 复盘目的

组织一次聚焦复盘：为什么当前 Energy Management iOS MVP 的体验与用户预期存在差异，并将差异转化为更清晰的 V2 产品方向和可验证的交互原型目标。

这次复盘不是为了证明当前实现是对的，而是为了把「哪里不对」说清楚，让下一轮设计、原型和 iOS 实现有明确目标。

## 当前初步判断

当前 App 基本完成了上一版「睡眠教练」方案的技术目标，但上一版方案可能把最初的产品意图收窄得太厉害。

初步判断的差异点：

- **产品定位差异：** 最初表达是「精力管理」，但当前 MVP 更像「睡眠作息仪式教练」。睡眠被处理成了整个产品，而不是每日精力管理的第一个锚点。
- **体验节奏差异：** 当前 App 是按页面完成任务：设置、今日节律、睡前指导、起床确认、报告。它缺少一个有辨识度的每日体验闭环。
- **设计质感差异：** 当前颜色方向符合暖灰米白，但视觉系统仍然太稀薄，离「简洁但是非常有设计感、有美感」还有距离。
- **报告价值差异：** 当前报告在数据口径上是正确的，但可能不够「精美」或不够有个人价值。它在汇报数据，但还没有讲出关于恢复、精力和次日状态的故事。
- **晨间体验差异：** 当前起床提示正确但偏通用，还不像一个能真正帮用户从困倦切换到清醒状态的晨间引导。
- **睡前仪式差异：** 当前有睡前建议，但缺少氛围、过程感和完成后的闭环反馈。
- **产品价值差异：** 当前产品帮助用户遵守睡眠纪律，但还没有回答更强的问题：「我如何管理和恢复我的精力？」

## 复盘原则

- 具体判断屏幕、时刻、文案和结果，不只讨论抽象喜好。
- 区分「产品定位问题」和「视觉设计问题」。
- 不因为体验不满意就盲目扩大功能范围；有些问题应该靠体验设计解决，而不是堆功能。
- 在重做 iOS 之前，优先做可点击的交互原型。
- 把当前 iOS App 视为学习样本，而不是失败结果。

## 复盘议程

### 1. 预期回放

目标：找回用户脑海里原本想要的产品。

需要回答：

- 你说「精力管理 iOS app」时，想象的是睡眠为主的 App，还是睡眠只是精力管理的一部分？
- 你期待打开 App 第一屏时的感觉更接近哪一种：安静的睡眠助手、每日精力仪表盘、私人教练、生活仪式空间，还是精美报告入口？
- 早上打开 App 后，你希望它让你产生什么感觉？
- 看到睡眠报告后，你希望它让你产生什么感觉？
- 当前哪个页面最明显不符合预期？
- 当前哪个页面最接近可接受？

### 2. 差异评分

每个维度按 1 到 5 分评分：

- 产品定位
- 视觉美感
- 高级感
- 交互顺滑度
- 情绪共鸣
- 睡前准备的有用性
- 早晨唤醒的有用性
- 报告的有用性
- 中文文案气质
- 每天想使用的欲望

低于 4 分的维度，都记录一个具体原因。

### 3. V2 产品方向选择

从以下三个方向中选择一个：

#### 方向 A：睡眠仪式

产品仍然以睡眠为核心，但变得更有氛围、更精美、更有仪式感，报告也更打磨。

适合情况：主要不满来自视觉质感和睡前/起床仪式感不足。

#### 方向 B：每日精力节律

睡眠仍然是基础，但产品表达变成每日精力节律：夜间恢复、晨间唤醒、白天精力预期、晚间复盘。

适合情况：主要不满是当前产品太窄，不像「精力管理」。

#### 方向 C：恢复报告

产品以精美报告为核心。设置和仪式较轻，核心价值是每天给出高质量的恢复/精力报告。

适合情况：你最期待的高光时刻是「报告很精美、很有洞察」。

当前推荐假设：**方向 B：每日精力节律**。

原因：它最贴近最初的「精力管理」表达，同时又能复用当前已经实现的睡眠基础。

### 4. V2 产品契约

进入实现前，需要明确：

- 一句话产品定位。
- 核心每日闭环。
- 首次设置范围。
- 夜间体验。
- 早晨体验。
- 报告体验。
- 视觉方向。
- 明确不做什么。
- 可以通过原型验证的验收标准。

### 5. 原型决策

推荐路径：

1. 写一版短 V2 PRD。
2. 做一个交互式 Web MVP 原型。
3. 用户体验原型并反馈。
4. 原型确认后，再更新 iOS V2 实现计划。

Web MVP 应该是可点击、有体验完成度的产品原型，不是 Landing Page。

## V2 假设方向

暂定名：**Energy Rhythm / 精力节律**

一句话定位：

> 一个极简、温暖、有设计感的每日精力节律助手，从睡前恢复、晨间唤醒到每日精力报告，帮用户把睡眠变成可感知的精力管理系统。

核心每日闭环：

1. 晚间：App 进入安静的恢复模式。
2. 睡前：用户收到温和的准备提醒。
3. 早晨：用户确认起床，并完成一个短促的唤醒仪式。
4. 报告：App 生成一份精美的恢复与精力节律报告。
5. 调整：用户获得一个很小、很明确的今晚或明早建议。

相对当前 MVP 的关键变化：

- 从静态的「今日节律」
- 变成被引导的「今日精力节律」

## Web MVP 范围

交互式 Web MVP 应包含这些可点击页面：

- 首次设置
- 今日精力节律首页
- 晚间恢复模式
- 睡前准备仪式
- 早晨起床确认
- 晨间唤醒仪式
- 每日精力/睡眠报告
- 七日节律报告

原型只需要本地内存状态，不需要真实通知、持久化、HealthKit、账号或后端。

## Web MVP 设计方向

风格：

- 极简日式 / 北欧
- 暖灰、米白、纸白、低饱和鼠尾草绿、柔和炭黑
- 安静但有高级感
- 比当前 iOS App 有更明确的视觉节奏
- 减少通用卡片堆叠，强化页面层级
- 把报告做成产品高光时刻

避免：

- 临床睡眠监测感
- 通用习惯打卡工具感
- 生产力仪表盘的拥挤感
- 无意义的装饰
- 过空、像未完成稿的页面

## 需要用户回答的问题

进入 V2 PRD 前，请先回答：

1. V2 应该是 **睡眠优先**，还是 **精力管理优先**？
2. 哪个最重要：**睡前仪式的体验感**、**早晨醒来的清醒感**，还是 **报告的精美和洞察**？
3. 当前最大失望点是什么：**定位不对**、**设计不够美**、**流程太弱**、**报告没价值**，还是 **文案不对味**？
4. 你希望 App 更像 **工具**、**生活方式产品**，还是 **私人教练**？
5. 报告应该更偏 **温柔叙事型**，还是 **数据清晰型**？
6. 「极简日式/北欧，暖灰米白」是否仍然是正确方向？还是应该更高级、更自然，或更未来感？

建议下一步走：

- 方向 B：每日精力节律

这样可以最大程度降低下一轮继续偏离预期的风险。
