# 产品文档：睡眠教练 V1.1

## 1. 结论

V1.1 的产品方向是：**一个极简、温暖、有仪式感的睡眠教练**。

它不做完整精力管理，不做医学睡眠追踪，不接入 HealthKit。它只解决一件事：帮助用户建立睡前准备、睡觉确认、起床确认、第二天回看报告这一条稳定动线。

当前 Web 原型已经是本版本的产品体验基准：

- 原型目录：`docs/prototypes/sleep-coach-web-v1-1`
- 本地地址：`http://localhost:5174`
- 高保真状态图：`docs/prototypes/sleep-coach-web-v1-1/screenshots/`

本阶段不再单独重画 UI 高保真图。原因是 Web 原型已经具备高保真视觉、文案、动效和关键状态，直接以原型截图作为设计状态图更贴近实现。

## 2. 背景

第一版 App 的问题不是“睡眠教练”定位错误，而是产品体验没有完成闭环：

- 设置只能第一次完成，之后无法自然修改。
- 睡前没有明确的「我睡觉了」按钮。
- 起床后没有明确的「我起床了」按钮。
- 睡前准备提醒和准备页没有形成可感知流程。
- UI 视觉质感不足，没有达到“极简日式/北欧、偏暖灰米白”的期待。

V1.1 通过 Web 原型重新定义用户动线和视觉标准，再回到原生 iOS 实现。

## 3. 产品定位

一句话：

> 睡眠教练帮助用户在晚上安静地结束一天，在早晨清醒地开始一天，并用一份克制、温暖的报告看见自己的睡眠节律。

产品语气：

- 少解释功能，多引导状态。
- 不强调“已记录”，记录应在后台完成。
- 不做任务清单压迫用户。
- 睡前场景强调平静、安宁、收束。
- 起床场景强调清醒、明亮、唤起。
- 报告强调“睡眠窗口”和“节律”，不做医学睡眠质量判断。

## 4. 目标用户

主要用户：

- 想建立稳定作息，但不想使用复杂习惯打卡工具的人。
- 睡前容易被屏幕和杂事拖住，需要一个轻柔提醒的人。
- 早晨醒来后容易刷手机，需要一个简单起床确认和唤起动作的人。

不是主要用户：

- 想看深睡、浅睡、REM 等医学或设备测量数据的人。
- 想使用强闹钟、贪睡、重复叫醒的人。
- 想管理白天精力、运动、饮食、工作效率的人。

## 5. V1.1 成功标准

- 用户第一次打开能在 1 分钟内完成睡觉时间、起床时间、睡前准备提前量设置。
- 用户在首页能随时修改作息。
- 睡前准备页直接提供「我睡觉了」按钮，不出现「我开始准备了」这类中间确认。
- 睡前准备页优先给出有科学依据的助眠 tips。
- 点击「我睡觉了」后，出现平静安宁的结束感。
- 起床页提供明确的「我起床了」按钮。
- 点击「我起床了」后，出现朝气蓬勃的唤起感。
- 今日报告基于用户手动点击的睡觉和起床确认生成睡眠窗口。
- iOS 实现通过现有单元测试、UI 测试，并在模拟器完成关键路径验收。

## 6. 范围

### 6.1 本版本包含

- 首次启动介绍页。
- 首次作息设置。
- 作息修改。
- 睡前准备提前量设置。
- 提醒开关和本地提醒。
- 今日睡眠首页。
- 睡前准备页。
- 睡觉确认。
- 睡觉确认后的安静结束动效。
- 起床确认页。
- 起床确认。
- 起床确认后的唤起动效。
- 今日睡眠报告。
- 七日节律趋势的轻量展示。

### 6.2 本版本不包含

- HealthKit。
- Apple Watch。
- 睡眠阶段分析。
- 医学睡眠评分。
- 强闹钟、贪睡、重复叫醒。
- 账号系统。
- 云同步。
- 分享长图。
- 订阅付费。
- 白天精力管理。
- 情绪、咖啡因、运动等复杂记录。

## 7. 核心动线

```text
首次打开
  ↓
建立一个安静、稳定的睡眠节律
  ↓
设置睡觉时间 / 起床时间 / 睡前准备提前量 / 提醒
  ↓
今日睡眠首页
  ├─ 修改作息
  ├─ 睡前准备 → 我睡觉了 → 可以安心睡了
  ├─ 起床确认 → 我起床了 → 开始清醒
  └─ 查看报告 → 今日睡眠报告
```

## 8. 关键页面

### 8.1 首次进入

目标：让用户理解睡眠节律的价值，而不是介绍功能列表。

核心文案：

- 标题：建立一个安静、稳定的睡眠节律。
- 正文：好的睡眠节律，会让身体更容易恢复，也让第二天醒来时多一点清醒和掌控感。
- 主按钮：开始设置。

### 8.2 设置作息

目标：让用户完成最少必要设置。

字段：

- 睡觉时间，默认 23:30。
- 起床时间，默认 07:30。
- 睡前准备，默认提前 45 分钟，可选 15 / 30 / 45 / 60 / 90 分钟。
- 提醒开关，默认开启。

### 8.3 今日睡眠首页

目标：让用户快速看到今天的睡眠节律，并能进入关键动作。

必须可见：

- 睡觉时间。
- 起床时间。
- 睡前准备开始时间。
- 修改作息。
- 睡前准备入口。
- 起床确认入口。
- 查看报告入口。

状态图：

![今日睡眠](../prototypes/sleep-coach-web-v1-1/screenshots/03-home.png)

### 8.4 睡前准备

目标：到睡前准备时间后，引导用户进入更容易入睡的状态。

信息顺序：

1. 当前状态：距离睡觉时间还有多少分钟。
2. 科学助眠 tips。
3. 直接确认按钮：「我睡觉了」。

科学 tips：

- 降低光线刺激：睡前减少屏幕和强光，让大脑更容易接收到夜晚信号。
- 让卧室安静、偏凉、偏暗：稳定的睡眠环境，比临时补救更能帮助身体进入休息。
- 避开临睡前刺激：尽量远离咖啡因、酒精、大餐和激烈运动，把最后一段时间留给放松。

状态图：

![睡前准备](../prototypes/sleep-coach-web-v1-1/screenshots/04-bedtime-preparation.png)

### 8.5 睡觉确认后

目标：让用户感到这一天已经平静结束。

行为：

- 用户点击「我睡觉了」。
- 系统在后台记录睡觉确认时间。
- 页面进入安静收束动效。
- 不突出“已记录”。

核心文案：

- 标题：可以安心睡了。
- 正文：今天到这里就好。把手机放远一点，让身体慢慢进入休息。

### 8.6 起床确认

目标：早晨给用户一个清晰、低阻力的起床确认。

核心动作：

### 8.7 起床确认后

- 系统在后台记录起床确认时间。
- 页面出现明亮唤起动效。
- 给出 3 个轻量行动提示。

提示：

- 喝几口水。

## 9. 用户故事与验收标准

### US-001：首次设置睡眠节律

作为用户，我想快速设置睡觉和起床时间，这样 App 可以基于我的节律提醒我。

验收标准：

- 首次打开显示介绍页。
- 点击「开始设置」进入设置页。
- 设置页包含睡觉时间、起床时间、睡前准备提前量、提醒开关。
- 点击保存后进入今日睡眠首页。
- 首页显示保存后的睡觉时间、起床时间和睡前准备开始时间。

### US-002：修改作息

作为用户，我想在首页随时修改作息，而不是只能第一次设置。

验收标准：

- 首页有可见的「修改作息」入口。
- 修改页复用当前已保存值。
- 保存后回到首页。
- 首页展示更新后的时间。
- 本地提醒按新时间重新调度。

### US-003：睡前准备提醒

作为用户，我希望到睡前准备时间时被轻轻提醒，这样我能开始远离屏幕并调整环境。

验收标准：

- 开启提醒时，系统每天调度睡前准备通知。
- 睡前准备通知时间 = 睡觉时间 - 提前量。
- 用户点击通知进入睡前准备页。
- App 内也能从首页进入睡前准备页。

### US-004：睡前准备与睡觉确认

作为用户，我想在准备好上床时点击「我睡觉了」，这样 App 能用我的真实确认生成报告。

验收标准：

- 睡前准备页先显示助眠 tips。
- 页面不出现「我开始准备了」按钮。
- 页面有醒目的「我睡觉了」主按钮。
- 点击后进入睡觉确认后页面。
- 睡觉确认时间被保存到当天记录。
- 确认后页面有平静安宁的收束动效。

### US-005：起床确认

作为用户，我想在早晨点击「我起床了」，这样我能开始清醒并完成睡眠记录。

验收标准：

- 起床确认页有醒目的「我起床了」主按钮。
- 点击后进入起床确认后页面。
- 起床确认时间被保存到当天记录。
- 确认后页面有明亮唤起动效。
- 页面给出喝水、拉窗帘、活动一分钟等轻量提示。

### US-006：今日报告

作为用户，我想看到基于手动确认生成的睡眠报告，这样我能理解自己的睡眠节律。

验收标准：

- 同一天有睡觉确认和起床确认时，报告显示睡眠窗口。
- 报告显示昨晚睡觉、今早起床、目标睡觉、目标起床。
- 报告文案明确这是手动确认形成的睡眠窗口，不是医学睡眠时长。
- 七日节律展示近期趋势。
- 缺少确认时，报告显示数据不完整状态，不伪造完整睡眠窗口。

## 10. 功能需求

- FR-1：系统必须保存一份当前作息设置，包括睡觉时间、起床时间、睡前准备提前量、提醒开关。
- FR-2：系统必须允许用户在首次设置后继续修改作息。
- FR-3：系统必须根据作息设置调度本地睡前准备提醒。
- FR-4：系统必须根据作息设置调度本地起床提醒。
- FR-5：睡前准备页必须直接提供「我睡觉了」按钮。
- FR-6：点击「我睡觉了」必须记录当天睡觉确认时间。
- FR-7：点击「我睡觉了」后必须展示平静结束状态。
- FR-8：起床确认页必须提供「我起床了」按钮。
- FR-9：点击「我起床了」必须记录当天起床确认时间。
- FR-10：点击「我起床了」后必须展示清醒唤起状态。
- FR-11：报告必须优先使用手动确认时间计算睡眠窗口。
- FR-12：报告不得声称测量了真实睡眠质量或睡眠阶段。
- FR-13：所有核心页面必须支持简体中文。
- FR-14：减少动态效果系统设置开启时，动效不得成为理解页面的唯一方式。

## 11. 数据与报告规则

### 11.1 记录项

每一天最多形成一条睡眠记录：

- 日期。
- 目标睡觉时间。
- 目标起床时间。
- 睡前准备提前量。
- 手动睡觉确认时间，可为空。
- 手动起床确认时间，可为空。
- 记录完整性状态。

### 11.2 睡眠窗口

睡眠窗口 = 起床确认时间 - 睡觉确认时间。

跨午夜场景需要正确处理，例如：

- 睡觉确认：23:28。
- 起床确认：07:34。
- 睡眠窗口：8 小时 6 分钟。

### 11.3 缺失数据

- 缺少睡觉确认：报告显示“缺少睡觉确认”。
- 缺少起床确认：报告显示“缺少起床确认”。
- 两者都缺失：报告显示“待完整”，不生成睡眠窗口。

## 12. 通知规则

- 睡前准备通知：每天在 `bedtime - prepLead` 触发。
- 起床通知：每天在 `wake` 触发。
- 用户修改作息后，应取消旧通知并重新调度。
- 用户关闭提醒后，不再调度未来通知。
- 通知点击路由：
  - 睡前准备通知 → 睡前准备页。
  - 起床通知 → 起床确认页。
  - 状态过期或无法判断 → 今日睡眠首页。

## 13. 文案规则

- 不在关键场景强调“已记录”，记录是系统后台行为。
- 不使用英文标签作为用户可见内容。
- 不使用功能说明式文案解释页面怎么用。
- 睡前文案保持短、慢、安静。
- 早晨文案保持清醒、明亮、行动导向。
- 报告文案保持克制，不做医学判断。

## 14. 风险与开放问题

- 首页目前仍偏工具入口，后续可以考虑根据当前时间窗口让主行动更突出。
- 报告当前是轻量趋势，后续要决定更偏数据分析还是更偏温暖反馈。
- Native iOS 需要在保留现有测试的同时重做 UI 质感，避免只复制功能、不复制体验。
- 本地通知在模拟器上可以验证调度和路由，但真实通知体验仍需真机体验。
