# PRD：睡眠教练 V1.1 体验修正版

## 1. 背景与问题

第一版产品定位保持为 **睡眠教练**，不扩展成完整的精力管理系统。当前版本的问题不是定位错误，而是两个核心体验没有达到预期：

1. **UI 高级感不足。** 当前界面虽然使用暖灰米白，但视觉层级、版式、组件质感、状态表达都偏基础，距离「简洁但是非常有设计感、有美感」还有明显差距。
2. **关键用户动线不可用。** 当前 App 只能首次设置作息，之后无法修改；睡前没有明确的「我睡觉了」确认按钮；起床后没有符合预期的「我起床了」确认按钮；睡前准备时间到达时没有有效提示出来。

V1.1 的目标是修正第一版作为睡眠教练的核心体验，让它先成为一个可用、好看、有仪式感的睡眠教练 MVP。

## 2. 产品定位

一句话定位：

> 一个极简、温暖、有设计感的睡眠教练，帮助用户在睡前进入准备状态，在上床睡觉和早晨起床时完成轻量确认，并生成一份优雅的睡眠节律报告。

本版本仍然只围绕睡眠，不引入白天精力管理、情绪记录、咖啡因、运动、HealthKit 或复杂习惯系统。

## 3. V1.1 目标

- 用户可以随时修改睡觉时间、起床时间和睡前准备提前量。
- 到达睡前准备时间时，App 能通过本地通知提醒用户开始睡前准备。
- 睡前准备页面有明确动线，并最终提供「我睡觉了」按钮。
- 起床时间到达后，App 提供明确、醒目的「我起床了」按钮。
- 报告能基于用户手动点击的「睡觉了」和「起床了」生成更符合直觉的睡眠记录。
- UI 从基础 SwiftUI 页面提升为有高级感的极简日式/北欧睡眠教练界面。
- 所有关键动线能在 iOS 模拟器中完整验证。

## 4. 非目标

- 不做完整精力管理系统。
- 不接入 HealthKit。
- 不接入 Apple Watch。
- 不做真实睡眠阶段分析。
- 不做医学睡眠质量判断。
- 不做账号、云同步、社交分享、订阅、导出。
- 不做复杂打卡系统。
- 不把睡前准备做成必须完成的任务清单。
- 不做闹钟替代品，不承诺强唤醒能力。

## 7. 用户故事

### US-001：用户可以随时修改作息

**描述：** 作为用户，我想在首次设置后仍然可以修改睡觉时间、起床时间和睡前准备提前量，这样当作息变化时 App 仍然可用。

**验收标准：**

- [ ] 首页存在明确的「修改作息」入口。
- [ ] 点击「修改作息」后展示当前已保存的睡觉时间、起床时间和睡前准备提前量。
- [ ] 用户保存修改后，首页立即显示新的作息。
- [ ] 保存修改后，旧提醒被取消，新提醒被重新计划。
- [ ] 重新启动 App 后，修改后的作息仍然存在。
- [ ] 使用 Build iOS Apps/XcodeBuildMCP 在模拟器中验证完整流程。

### US-002：睡前准备时间到达时提醒用户

**描述：** 作为用户，我想在睡前准备时间到达时收到提醒，这样我能开始远离屏幕、降低灯光并准备睡觉。

**验收标准：**

- [ ] 保存作息后，系统为每天的睡前准备时间创建本地通知。
- [ ] 通知标题和内容明确表达「开始睡前准备」。
- [ ] 点击睡前准备通知后进入睡前准备页面。
- [ ] 如果通知权限未开启，首页显示低干扰提醒，并说明需要开启通知才能收到睡前准备提醒。
- [ ] 修改作息后，睡前准备通知时间同步更新。
- [ ] 使用模拟器或可控测试注入验证通知路由。

### US-003：用户可以在睡前点击「我睡觉了」

**描述：** 作为用户，我想在真正准备睡觉时点击「我睡觉了」，这样 App 能记录我的手动睡觉时间，并用于第二天报告。

**验收标准：**

- [ ] 睡前准备页面直接显示明确的「我睡觉了」主按钮。
- [ ] 睡前准备页面不提供「我开始准备了」按钮，也不要求用户先确认开始准备。
- [ ] 点击「我睡觉了」后记录当天睡觉确认时间。
- [ ] 点击后展示已记录状态，而不是让用户不确定是否成功。
- [ ] 同一天重复点击时，不应生成重复记录；可以展示已记录状态或允许用户更正。
- [ ] 报告中展示手动确认的睡觉时间。
- [ ] 文案不得声称 App 测量了真实入睡时间，只能表达为「睡觉确认时间」或「上床记录」。
- [ ] 使用 Build iOS Apps/XcodeBuildMCP 在模拟器中验证完整流程。

### US-004：用户可以在起床后点击「我起床了」

**描述：** 作为用户，我想早上起床后点击「我起床了」，这样 App 能确认我已经醒来，并给我简短的清醒提示。

**验收标准：**

- [ ] 起床时间窗口内，首页或起床确认页显示明确的「我起床了」主按钮。
- [ ] 点击「我起床了」后记录当天起床确认时间。
- [ ] 点击后展示晨间唤醒提示，例如喝水、拉开窗帘、站起来活动。
- [ ] 起床确认后可以进入今日报告或回到首页。
- [ ] 错过确认窗口时，报告中应明确标记为未确认或估算，不应默认为成功起床。
- [ ] 使用 Build iOS Apps/XcodeBuildMCP 在模拟器中验证完整流程。

### US-005：报告基于手动睡觉和起床确认生成

**描述：** 作为用户，我想看到基于我实际点击记录生成的睡眠报告，这样报告更符合我对「我几点睡、几点起」的直觉。

**验收标准：**

- [ ] 当存在「我睡觉了」和「我起床了」记录时，报告显示两者对应时间。
- [ ] 报告显示睡眠窗口，例如 8 小时 6 分钟。
- [ ] 报告显示相对目标作息的偏差，例如晚睡 2 分钟、晚起 4 分钟。
- [ ] 如果缺少睡觉确认或起床确认，报告必须明确说明数据不完整。
- [ ] 报告中保留估算边界说明，不做医学或睡眠阶段判断。
- [ ] 使用单元测试覆盖完整记录、缺少睡觉记录、缺少起床记录、错过起床确认等情况。

### US-006：UI 达到更高的设计质感

**描述：** 作为用户，我希望 App 看起来简洁、高级、温暖、有设计感，这样我愿意每天使用它，而不是把它当成一个粗糙工具。

**验收标准：**

- [ ] 首页、睡前准备、起床确认、报告四个核心页面完成统一视觉升级。
- [ ] 视觉方向保持极简日式/北欧、暖灰米白。
- [ ] 页面层级更清晰，核心动作更突出。
- [ ] 减少通用卡片堆叠，增加更有节奏的空间、字号和信息组织。
- [ ] 按钮、时间选择器、报告模块具备一致的高级感。
- [ ] 在至少一个 iPhone 模拟器截图中验证首屏没有拥挤、错位或廉价感。

## 8. 功能需求

- **FR-1：** App 必须支持首次设置睡觉时间、起床时间和睡前准备提前量。
- **FR-2：** App 必须支持首次设置后的作息修改。
- **FR-3：** 修改作息保存后，首页、报告计算和通知计划必须同步更新。
- **FR-4：** App 必须在睡前准备时间创建本地通知。
- **FR-5：** 点击睡前准备通知必须进入睡前准备页面。
- **FR-6：** 睡前准备页面必须提供具体但非强制的睡前建议。
- **FR-7：** 睡前准备页面必须直接提供明确的「我睡觉了」按钮，不设置「我开始准备了」中间按钮。
- **FR-8：** 点击「我睡觉了」必须记录当天睡觉确认时间。
- **FR-9：** App 必须在起床时间附近创建起床提醒。
- **FR-10：** 起床确认页面必须提供明确的「我起床了」按钮。
- **FR-11：** 点击「我起床了」必须记录当天起床确认时间。
- **FR-12：** 起床确认后必须展示简短晨间唤醒提示。
- **FR-13：** 报告必须优先使用手动睡觉确认和起床确认生成睡眠窗口。
- **FR-14：** 缺少手动确认时，报告必须明确标记为未确认或估算。
- **FR-15：** App 必须处理通知权限未开启状态。
- **FR-16：** App 必须处理错过起床确认窗口状态。
- **FR-17：** App 必须保持本地数据，不依赖网络、账号或 HealthKit。
- **FR-18：** 核心页面必须符合 44pt 最小触控目标、Dynamic Type 和 VoiceOver 基础要求。

## 9. 设计要求

### 9.1 视觉方向

- 极简日式/北欧。
- 暖灰、米白、纸白为主。
- 使用低饱和自然色作为少量强调，例如鼠尾草绿、雾蓝、柔和炭黑。
- 页面应安静、克制，但不能空到像未完成。
- 字体层级要明显，时间和关键动作要有产品感。

### 9.2 核心页面设计重点

首页：

- 让用户一眼知道今天的睡眠状态。
- 修改作息入口必须明确。
- 当前最重要动作必须突出，例如睡前准备、我睡觉了、我起床了。

睡前准备页：

- 有夜间氛围，但不沉重。
- 建议应该像引导，不像任务清单。
- 「我睡觉了」是最终动作。

起床确认页：

- 清爽、明亮、直接。
- 「我起床了」按钮必须非常明确。
- 起床后给出 2 到 3 个简单动作，不要长篇解释。

报告页：

- 是产品高光页面。
- 应该美观、清晰、容易截图。
- 优先展示：睡觉确认时间、起床确认时间、睡眠窗口、与目标偏差、今晚建议。

## 10. 数据口径

V1.1 可以使用用户手动确认数据，但必须注意口径：

- 「我睡觉了」记录的是用户点击确认的上床/睡觉时间，不等于真实入睡时间。
- 「我起床了」记录的是用户点击确认的起床时间，不等于生理醒来时间。
- 「睡眠窗口」是两个手动确认之间的时间窗口，不是医学意义上的睡眠时长。
- 报告可以使用「睡眠窗口」「作息信号」「确认记录」，避免使用「深睡」「睡眠质量分」等医学化表达。

## 11. 技术考虑

- 继续使用本地数据存储。
- 当前 `SleepSchedule` 需要支持编辑后重新调度通知。
- 当前 `SleepRecord` 需要明确支持 sleep confirmation 和 wake confirmation。
- 通知调度服务需要可测试：保存作息、修改作息、权限关闭、通知点击路由都需要覆盖。
- 时间相关逻辑应使用可注入 clock/date provider，便于模拟睡前准备时间、睡觉时间和起床时间。
- UI 验证应使用 XcodeBuildMCP/Build iOS Apps 跑测试，并配合模拟器截图检查。

## 12. 成功指标

V1.1 通过标准：

- 用户能在 30 秒内完成首次设置。
- 用户能在首页 1 次点击进入修改作息。
- 到达睡前准备时间时，通知能出现并正确进入睡前准备页。
- 用户能完成「睡前准备 → 我睡觉了 → 次日起床 → 我起床了 → 查看报告」完整动线。
- 报告能展示手动确认的睡觉时间和起床时间。
- 首页、睡前、起床、报告四个核心页面的视觉明显优于当前版本。
- 所有现有单元测试和 UI 测试通过，并补充 V1.1 新动线测试。
